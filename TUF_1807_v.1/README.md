# TideUp File (TUF) — v0.41

## Avvio (sviluppo)
```
pip install -r requirements.txt
python -B -m filepilot.main
```
Il flag `-B` dice a Python di non usare MAI la cache compilata (`__pycache__`), cosi' e' certo che ogni volta
gira il codice appena estratto dallo zip, anche se una versione precedente ha lasciato una cache in giro
nella stessa cartella. Se una modifica sembra "non essere arrivata" nonostante tu abbia estratto l'ultimo
zip, cancella anche eventuali cartelle `__pycache__` rimaste da versioni precedenti ed usa `-B`.

## Novità v0.27

- **La finestra non riparte piu' dalla posizione/dimensione salvata.** Il fix della v0.26 non ha risolto il problema (log del terminale quasi identico anche dopo, inviato di nuovo da un utente): la raffica di "QWindowsWindow::setGeometry" compare SEMPRE appena il programma parte, prima di qualunque azione, con valori quasi identici run dopo run — troppo deterministico per essere causato dall'interfaccia durante l'uso. Il sospetto piu' forte e' la geometria SALVATA che veniva ripristinata all'avvio (restoreGeometry): se risulta scomoda per Qt su un monitor/scaling specifico, Windows la rinegozia allo stesso modo ad ogni riavvio. Invece di continuare a inseguire il sintomo, si toglie la causa: ora la finestra riparte sempre dalla stessa dimensione di default (1400x900), non piu' da quella salvata dell'ultima sessione. Si perde il "riapre dov'era rimasta", ma si guadagna un avvio sempre affidabile. Splitter, ordinamento e cartelle restano comunque salvati come prima — solo la posizione/dimensione della finestra non viene piu' ripristinata. ## Correzioni in questa versione
- **La barra di stato dei duplicati restava sempre ferma a 1 file dalla fine**: causa trovata. La fase finale di conferma (hash completo sui candidati sospetti) non emetteva mai un aggiornamento di progresso, quindi se gli ultimi file richiedevano una conferma lunga sembrava tutto fermo anche se stava ancora lavorando. Ora aggiorna la barra di stato anche durante questa fase
- **"Elimina copie" lento a ricaricare**: le miniature di foto e PDF venivano ridecodificate da zero ad ogni ricostruzione della lista (solo i video avevano una cache). Ora tutte le miniature sono in cache, quindi ricostruire la lista dopo un'azione e' quasi istantaneo
- **Blocco in apertura file, ancora**: trovata un'altra causa concreta. Le correzioni precedenti rendevano asincrona la decodifica, ma ogni nuova richiesta partiva subito con un thread nuovo: navigando molto velocemente tra tanti file grandi (es. scorrendo la lista ordinata per dimensione) i thread di decodifica potevano accumularsi ed esaurire di nuovo la CPU. Ora c'e' sempre al massimo un thread di decodifica attivo: le richieste in eccesso si mettono in coda invece di partire tutte insieme
- **Blocco ordinando per dimensione — trovato il pezzo mancante**: la correzione precedente rendeva asincrone foto e PDF, ma il poster (l'anteprima) dei VIDEO era rimasto sincrono per una svista. Se il file piu' grande della cartella e' un video (molto comune: i video pesano piu' delle foto), il blocco si ripresentava identico. Ora anche il poster video si carica in un thread separato, come foto e PDF
- **"Non risponde" ordinando per dimensione**: causa trovata. Ordinando per dimensione, il file PIU' GRANDE della cartella diventa quello mostrato — se e' un PDF complesso o una foto molto pesante, decodificarlo poteva richiedere parecchio tempo, e prima questo avveniva in modo sincrono bloccando tutta l'interfaccia nel frattempo. - **"Non risponde" durante la scansione duplicati se si tocca la lista**: cambiare l'ordinamento della lista file (Nome/Dimensione/Data) MENTRE la scansione duplicati gira in sottofondo poteva far sembrare l'app bloccata: due lavori pesanti sulla CPU insieme (l'hash dei duplicati + il riordino/ridisegno della lista) si contendono il tempo di Python, e per qualche secondo l'interfaccia non risponde (si sblocca sempre da sola, non e' un crash, ma sembra uno). Ora durante la scansione i controlli della lista (ordina/filtra/cerca) sono disattivati, cosi' non si puo' piu' innescare questa situazione — e si vede subito che l'app e' occupata a scansionare, non bloccata
- **Scansione duplicati lenta/"non risponde" vicino alla fine**: causata dalla correzione precedente (leggere l'intero file sotto i 200 MB): con 1800+ file era troppo lavoro, anche se non un vero crash — la finestra si sbloccava da sola dopo un po' ("non risponde" temporaneo di Windows, non un arresto definitivo). Ora la scansione usa un hash veloce campionato per TUTTI i file (come prima), e legge il file per intero SOLO per confermare i pochissimi candidati che hanno gia' stessa dimensione e stesso hash campionato — stessa sicurezza contro i falsi duplicati (bug dei CV), niente piu' rallentamento su librerie grandi
- **Trovata la vera causa del crash (grazie al log dal terminale)**: `ui/preview_widget.py` era rimasto TRONCATO a meta' di un'istruzione dentro `_update_type_badge` (mia svista in una modifica precedente: un salvataggio interrotto a meta'). Il file passava comunque il controllo di sintassi (l'istruzione tagliata era comunque valida come sintassi, solo sbagliata a runtime), quindi non l'avevo notato. Il crash scattava ogni volta che, dopo aver chiuso la finestra duplicati, si arrivava a visualizzare un file non multimediale (PDF, Word, zip, ecc.) — non era quindi legato ai duplicati in se', ma capitava spesso subito dopo perche' e' li' che si torna chiudendo quella finestra. Le due correzioni precedenti (pulizia del widget della lista, niente scorrimento annidato per gruppo) restano comunque miglioramenti validi, ma non erano la causa di questo crash specifico — ora e' risolto per davvero, verificato riga per riga su tutti i file toccati
- **Altro crash nell'apertura dei duplicati — probabile causa trovata**: ogni gruppo di duplicati aveva una propria mini area di scorrimento orizzontale; con tanti gruppi insieme (bastano poche centinaia di file duplicati) questo creava altrettante aree di scorrimento native tutte insieme, ed e' un sospetto concreto per un crash all'apertura. Rimosse: ora i gruppi non hanno piu' scorrimento proprio (se un gruppo ha tanti file, scorre la finestra intera). Nota: non ho un log dell'errore, quindi non posso essere certo al 100% che fosse questa la causa — se dovesse ripresentarsi, vedi la nota sotto
- **Crash cambiando l'ordinamento dei duplicati (Crescente/Decrescente)**: risolto. Causa: ogni volta che si ricostruiva la lista, il widget precedente (con dentro le mini-aree di scorrimento per ogni gruppo) non veniva mai eliminato esplicitamente e restava "orfano" in memoria; prima o poi Qt andava in crash. Ora il widget vecchio viene tolto ed eliminato esplicitamente prima di costruire quello nuovo
- **Bug serio risolto — duplicati sbagliati**: due file diversi ma della stessa dimensione (es. due CV con la stessa dimensione ma una foto diversa nel mezzo) potevano essere segnalati come duplicati. Causa: l'hash di confronto leggeva solo il primo e l'ultimo blocco da 1 MB del file, ignorando il centro. Ora sotto i 200 MB l'hash legge l'intero file (zero rischio di falso duplicato); sopra i 200 MB (video molto grandi) campiona 8 punti distribuiti nel file invece di 2, molto piu' robusto
- **Rischio di blocco nell'apertura della finestra duplicati**: per i video, prima venivano aperti due volte (una per la miniatura, una per calcolare la durata). Ora si apre una volta sola per entrambe le cose, dimezzando il lavoro sincrono quando ci sono tanti duplicati video insieme
